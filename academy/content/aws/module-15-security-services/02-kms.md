# AWS KMS: Key Management and Encryption

## Overview

AWS Key Management Service (KMS) is the central key management service for AWS. It creates, stores, and manages cryptographic keys used by dozens of AWS services for encryption. Understanding KMS — envelope encryption, key policies, grants, and multi-region keys — is required for any AWS security architecture.

## KMS Key Types

AWS Managed Keys (aws/service): created automatically by AWS services (e.g., aws/s3, aws/rds), rotated annually, you cannot manage them directly — fine for basic encryption. Customer Managed Keys (CMK): you create them, control the key policy, can enable automatic annual rotation, and can audit every usage via CloudTrail. Use CMKs when you need control over who can use the key and to meet compliance requirements. Asymmetric keys (RSA, ECC) are also supported for signing and public-key encryption outside AWS.

## Envelope Encryption

KMS does not encrypt your data directly at scale — it uses envelope encryption. A Data Key is generated by KMS for each encryption operation. The data key encrypts the actual data (locally, in memory). The data key itself is then encrypted by the KMS CMK (the 'envelope'). Only the encrypted data key is stored alongside the ciphertext. To decrypt: send the encrypted data key to KMS, KMS decrypts it with the CMK, the plaintext data key is returned in memory (never stored by KMS), and you decrypt the data. This separates key management from data encryption and scales to large payloads.

## Key Policies and Grants

Every KMS CMK has a key policy — a resource-based policy that defines who can use and administer the key. Unlike IAM policies alone, IAM permission is insufficient for KMS — you also need an explicit Allow in the key policy. The default key policy allows the account root full control. Grant specific IAM roles key usage (kms:Decrypt, kms:GenerateDataKey) and separate key administration (kms:CreateKey, kms:PutKeyPolicy). Grants are programmatic delegations of key usage — used by AWS services to use a CMK on your behalf without modifying the key policy.

## Multi-Region Keys

Multi-Region KMS Keys are a set of interoperable keys in different regions with the same key material and key ID. Encrypt in us-east-1, decrypt in eu-west-1 — without re-encrypting. Use for: globally distributed applications that need consistent encryption, active-active multi-region architectures, or encrypting global DynamoDB tables. The primary key exists in one region; replica keys are synchronized copies.

## CloudTrail KMS Audit

Every KMS API call — including every Decrypt and GenerateDataKey — is logged in CloudTrail with the caller identity, source IP, key ARN, and timestamp. This is the audit trail proving who decrypted what data and when. For compliance-sensitive data, the CloudTrail KMS log is the evidence trail for access audits. Never use AWS Managed Keys if you need this level of audit visibility — you get CloudTrail for CMKs but not for aws-managed keys in the same detail.

## Summary

KMS provides centralized key management with full CloudTrail audit logging. Use CMKs for control and compliance. Envelope encryption scales KMS to large data volumes. Key policies and IAM together control key usage. Multi-Region Keys enable global encryption consistency. Every serious security architecture on AWS is built on KMS.
