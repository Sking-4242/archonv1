# Azure Monitor: The Central Nervous System

## Overview

In the on-premises world, IT departments bought separate software packages to monitor servers, network switches, and application performance. In Azure, telemetry is centralized into a single, massive service: **Azure Monitor**.

Azure Monitor is the equivalent of Amazon CloudWatch, but it is deeply embedded into the fabric of the Azure Resource Manager (ARM). It collects, analyzes, and acts on telemetry data from your cloud and on-premises environments. To master Azure Monitor for the AZ-104, you must understand the fundamental divide in how it handles data: **Metrics vs. Logs**.

## The Data Divide: Metrics vs. Logs

Azure Monitor ingests massive amounts of data, but it immediately sorts that data into two entirely different databases optimized for different purposes.

**1. Metrics (The "What")**
Metrics are numerical values collected at regular intervals. They are incredibly lightweight and highly structured.
* **Examples:** CPU percentage, memory available, disk read operations, network bandwidth.
* **Characteristics:** Because they are just numbers with timestamps, they are stored in a time-series database. This makes querying them almost instantaneous. Metrics are used for real-time alerting (e.g., "Trigger an alert if CPU > 80% for 5 minutes") and auto-scaling rules.

**2. Logs (The "Why")**
Logs are dense, complex records of events. They contain unstructured or semi-structured data (like JSON payloads or raw text).
* **Examples:** Windows Event Logs, Syslog from Linux, an application throwing a Null Reference Exception, or a user logging in from an unauthorized IP.
* **Characteristics:** Logs are heavy. You cannot use them to trigger a sub-second auto-scaling event. However, when CPU spikes to 100% (the *What*), you query the Logs to find out exactly which application process caused the spike (the *Why*).

## Activity Logs vs. Resource Logs

Within Azure, you must also distinguish between the control plane and the data plane. Azure Monitor captures both, but calls them different things.

* **The Azure Activity Log (Control Plane):** This is the direct equivalent of **AWS CloudTrail**. It records the "Who, What, and When" of every management action taken in your subscription. If a junior administrator deletes a production Virtual Machine, the Activity Log records their Entra ID username, the exact time, and the API call made. *Activity Logs are kept for 90 days by default.*
* **Resource Logs (Data Plane):** These are the logs generated by the resources themselves from the inside. For a SQL Database, it might be a log of slow queries. For a Key Vault, it is a log of who requested a secret. *Resource logs are not collected automatically; you must explicitly enable "Diagnostic Settings" to route them somewhere.*

## Summary

Azure Monitor is the central hub for all telemetry in your cloud environment. It splits data into fast, lightweight **Metrics** (used for real-time alerts and auto-scaling) and dense, complex **Logs** (used for deep troubleshooting). The Azure Activity Log tracks management actions (like AWS CloudTrail), while Resource Logs track the internal operations of the services themselves.
